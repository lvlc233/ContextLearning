# 上下文学习

## 前言

Agent 领域从 2023 年开始，已经发展和探索了许多内容。严格来说，Agent 领域目前主要研究的方向可以分为 3 类，它们分别对应 Agent 工程中的三个核心问题：

| 研究方向 | 代表内容 | 回答问题 |
| --- | --- | --- |
| Agent 运行环境 | sandbox、harness 等 | Agent 在哪运行？ |
| 运行引擎 | ReAct 范式、loop、dynamic workflows 等 | Agent 如何运行？ |
| 上下文管理 | CLAUDE.md、skill、memory 等 | Agent 要运行什么？ |

本文讨论的内容属于 **上下文管理** 中的 **上下文内容** 方向，因此不涉及 **上下文管理** 中的 **上下文操作** 方向的内容，也不展开 Agent 运行环境和运行引擎相关问题。

## 介绍

**上下文学习** 是本文中最基础、也最重要的一个概念。它源于 Agent 技术发展过程中被反复提及的工程命题，例如：`Agent 自进化`、`memory 进化`、`skill 自迭代` 等。后续出现的、以调整上下文内容来改变行为为核心的问题，也可以在这一范畴下继续讨论。

这个概念并不旨在成为过去或未来相关概念的竞争产品，而是提供一套标准范式，用来统一讨论这一领域中的问题，从而破除 **上下文学习** 领域中的巴别塔效应。

在本文中，有时会使用 **上下文训练** 代替 **上下文学习**。两者基本相同，只是前者有时更强调在 `训练` 阶段的过程。大部分情况下，两者可以视为等价。

## 定义

源于机器学习，上下文学习指的是：围绕显式或隐式的训练目标，在指定的上下文空间中，通过训练信号、训练方向、数据、标注对上下文信息进行持续调整，使模型在加载这份上下文后更稳定地逼近期望行为的过程。

训练目标不一定在训练开始时就已经完整明确。它可以先隐含在人类偏好、样本表现、失败现象或版本对比中，再在上下文训练过程中被逐步显式化。

上下文学习围绕一份上下文本体，通过非参数调整的方式，使模型在加载这份上下文后更稳定地逼近目标行为。

这里的 `上下文` 可以包括：

- system prompt
- skill
- memory
- plan creator
- tool info...

尽管这个训练目标可以不依赖于模型存在，但是绝大部分情况下，我们都希望这份上下文能够对模型本身产生效果。因此在后续讨论中，我们会经常使用“上下文影响模型……”等描述，但请注意，这并非必然因素。

## 基本元素

一次上下文学习至少需要区分几个元素：

- **上下文本体**：被训练或被修改的上下文，例如 skill、prompt、rule。可以类比为机器学习中的模型参数。
- **训练目标**：希望上下文影响模型后产生的行为方向。它可以是显式写出的目标，也可以先隐含在偏好、标注或失败现象中。
- **训练方向**：判断当前输出或上下文修改是否更接近训练目标的标准。它可以从偏好信号、失败案例或目标偏差中提炼出来，可类比为机器学习中的奖励信号。
- **训练信号**：运行、标注、测试或版本对比中产生的判断依据，用来帮助判断上下文是否正在逼近目标。
- **训练数据**：用于暴露问题、形成方向、进行训练或测试泛化的样本。详情参考 [xxx]。
- **反馈与标注**：对输出的判断、偏好和修正意见。例如“这段输出是合理的”“这段输出不错，但是感觉结构有点固定”“这段信息不太好”。
- **测试与 checkpoint**：用于比较版本、观察稳定性和防止漂移。

## 上下文学习类型

### 静态上下文学习

在当前阶段（2026/06），我们所提及的 `上下文学习` 通常指代 `静态上下文学习`。静态上下文指的是生成与使用不依赖运行状态的上下文信息，这些信息通常在 Agent 运行中作为相对固定的材料被加载，例如 Agent 运行的初始化行为。

例如 `skills`、`memory` 等，便属于 `静态上下文` 领域。

静态上下文训练关注的是：

```text
如何修改这份上下文本身，使它在加载时能够更稳定地使 Agent 运行对应的任务目标。
```

## 边界

上下文学习要处理的问题包括：

- 输出质量不稳定；
- 目标不清；
- 规则越改越漂移；
- 上下文信息冗余；
- 案例导致过拟合；
- 版本对比缺失；
- 人类偏好无法稳定转成 Agent 行为。

上下文学习不能保证找到最优规则。

它解决的是另一个问题：

```text
不要只靠感觉改 prompt。
不要把概念直接塞进 skill。
不要只看一个案例就以为变好了。
```

它让改造过程至少有目标、有材料、有测试、有反馈、有版本。
