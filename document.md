### 这是什么?
这是一套方法论,适用于任意的含上下文训练的任何内容形式;

它解决这样子的一组问题,即通过标准的、可量化、可复刻的科学训练标准过程将基于高等语言模型的生成内容质量低,过拟合,无量化与标准的感觉式训练不可靠转化为稳定进步的生成过程;

## 基本流程

可以把流程拆成六个步骤。

```text
→ 数据集准备
→ 设置训练流程与参数
→ 制订/提炼 训练目标 (参考目标函数/奖励信号)
→ 受人类标注[偏好]的前训练/数据标注(可选但推荐)
→ 训练
→ 受人类标注[偏好]的后训练(可选)
```

每个过程都可以独立的循环进行,也可以构建为一个更大的循环结构进行运行;

## 1. 数据集与准备
最好的数据集就是从实际的生成活动中产出,它可以来自你本人的会话;
并尝试根据不同的对话内容,进行粗略的数据标注,常见的单位有:
- 以一次 run 记录作为数据单位(通常去除工具调用等细节)
- 以一组与指定任务相关的 run 记录为数据单位
- 以一个完整会话作为单位

关于数据集合的类型:

有常见的
- 训练数据集
- 测试数据集
- 标注数据集
    [
        通过人类标注少量输入输出对的信号,让模型拥有更好的标准用于进行上下文学习监督
    ]
    - 偏好数据集
        [
            同属于标注数据集的子类,常见于特定风格偏好,如前端美术细节实现风格的标注信息等;
        ]

与:
- 训练目标集

## 2. 设计训练流程与参数
包括训练迭代方式,轮次,选择的数据集大小,批次,更改量,评估标准等可以理解为"超参数"的内容;(尽管被这么比喻,但实际上,它可以是一套自然语言描述,也可以是一套程序系统,例如loop)

一个好的训练训练过程与参数通常也不是一触即就,相反,它与上下文本身一样,需要不断的调试与迭代;

而一个好的训练流程与参数通常可以复用在许多不同场景的上下文训练过程中,希望您能保留下来属于自己的训练流程与参数

## 3. 制订/提炼 训练目标

好的答案源于好的问题,好的训练源于好的目标;

尽管在机器学习领域,制定一个好的学习目标十分困难,但是在上下文学习方向,使用自然语言描述却相对简单许多;

常见的训练目标就是从人类偏好中提取:
- 行为习惯
- 审美偏好
- 表达风格
...

从经验中提炼出规律并总结成概念,并赋予解释意义是常见的目标定义方式;

在定义训练目标的时候建议考虑训练目标的正交性,即互不干扰的训练方向,因为只有这样子才能够并发执行,并清晰的分清原因;

建立创建一个专属的目标集合文档,明确训练目标

### 建议
尽管上下文学习的训练目标门槛不算高,但是随随便便的一个目标也只会导致训练出一个模糊混乱的上下文;

一个建议是:

不要一开始就写规则。

先确认问题到底是什么。

例如这轮最早的问题不是：

```text
agent 不够苏格拉底。
```

而是：

```text
agent 在用户已经给出思路后，仍然机械提问；
没有先判断用户哪里对、哪里错；
问题有时候和用户当前注意力不对齐。
```

问题要尽量落到行为上。

坏的问题描述：

```text
反馈不好。
```

更好的问题描述：

```text
用户写了伪代码，但 agent 没指出具体哪一行的操作语义错了，而是转去问另一个问题。
```


## 4. 数据标注
有别于机器学习领域的数据标注,上下文学习因为语言模型具备自然语言理解能力,因此在标注的过程中,可以通过多元的自然语言方式进行标注;

即可以简单的对 待数据标注对象标记 [好/坏]的标签
也可以详细到对每个输入项,乃至"参数"(例如上下文本体/修改内容/...)进行描述性补充说明;

一个案例是:
- 哪一句好；
- 哪一句不好；
- 为什么不好；

事实上有别于传统机器学习中若不进行数据标注就无法进行有效的学习过程;大语言模型是具备一定能力将某段上下文进行优化的能力的,但是这种优化能力依赖于原始上下文信息与模型本身偏好,常见现象就是:
初始轮次优化尚可,但是多轮优化后优化方向和内容开始呈现随机漂移与无效化优化;
多次独立优化总能找到优化点,尽管这些优化点在其他地方被视为良好内容;
总的来说呈现出一种"为了修改而修改"的惯性运动;
而这也正是真动态自上下文优化的困难所在;

因此在这一步,为上下文学习提供数据就是给定模型一个清晰的训练方向;(而有别于目标)




例如：

```text
“这一轮只检查一件事情”会让人有被控制感，不像引导。
```

这不是单纯的差评，而是一个很具体的偏好信号。

它可以转成规则：

```text
不要用控制式表达来限制用户。
如果需要聚焦，说明聚焦理由，而不是命令用户只能看一个点。
```

## 5. 训练
此处训练特指"通过有方向性的指导,对模型上下文进行修改,并通过反馈信号逼近指定目标的过程"

因此一个最基础的训练过程如下
```text
→ 修改上下文
→ 放入模型中
→ 输入数据集
→ 跑测试
→ 评估偏离程度
```
跑输出时要注意保留原始结果,通过可以使用会话信息进行保留;

将训练的每个过程都展开观测,有助于后续调整与分析;

在训练的过程中至少注意保持
- 足够的训练样本空间(防止过拟合 + 加强训练底层而非表象)
- 平均有效的训练过程(避免偶发性幸运与失败破坏影响实际数据)
- 足够的训练时间(多次训练使信号逼近而不是期望一次训练就可以完成目标)
- 上下文局限性,对于同一个目标对于不同的起始条件(包括起始上下文)和推理模型来说,会有不同的表现;也就是说相同的上下文对于不同的模型来说不完全适配

### 技巧
训练时优先使用同系列轻量化模型。

因为同系列模型在训练时候的使用数据与架构更加接近,使用轻量化模型可以降低训练所需成本

但这个缩放有边界。

如果模型能力差距太大，可能会存在其他模型不兼容现象。


在训练过程中,单次的训练可以分为单一目标的训练或组合训练,并不建议一次性对所有的目标进行训练,因为对所有目标进行同期训练可能会导致训练有效性需要长周期与多测试验证才能证明效果稳定优秀;

如果了解大模型moe架构的可参考moe架构进行上下文训练;




## 6. 后训练过程

这一步实际上与前期数据标注在上下文学习领域中差距更小,本事上就是对训练完或正在训练的上下文空间进行偏好性微调,使用技巧和方式与前期标注和训练无本质上差距;

这个过程更像是一个人类专家/老师,在校准模型调整上下文空间的迭代方向,避免其偏离了训练目标;

也可以在这个过程中像是进行lora训练一样进行细节微调

在这个过程中,可以尝试直接从描述中令监督模型将原始输入转化为符合当前上下文空间的动作规则,因此直接嵌入偏好;

## checkpoint
该概念取自 模型训练 过程中的 训练结果保存

checkpoint 至少保留：

- 当前上下文版本号(用于标记)
- 上下文本体
- 训练时使用的数据集与输出(用于观测)
- 训练目标偏离程度(类似于loss)

### 使用git作为 checkpoint 存储工具

在进行 "训练" 过程中,可以选择git提交或tag记录每个checkpoint;

如果使用多维度分区训练的方式,可以使用文件夹做内容隔离;

推荐以checkpoint作为单体而非某次训练迭代作为提交单位



## 这套流程的边界

它不能保证找到最佳规则。

它解决的是另一个问题：

```text
不要只靠感觉改 prompt。
不要把概念直接塞进 skill。
不要只看一个案例就以为变好了。
```

它让改造过程至少有材料、有测试、有反馈、有版本。

这已经比单纯“我觉得这样写更好”稳定很多。

## 一些在训练过程中的反例信号

- 直接修正:直接将输入端的反馈信号作为待迭代上下文中某个区域;
- 案例优先:使用案例

这些信号统一的特征就是,并非以输入端的信号作为导向进行内容学习,而是直接将输入端的内容进行嵌入,从而导致三大问题:
1. 过拟合
2. 浅学习
3. 上下文污染



## 一些在训练过程中良好的训练信号

### 使用动作信号尝试表示/拟合某一个概念或需求

例如将:
```text
保持聚焦
```

转化为:

```text
每轮只处理一个主要问题。
如果用户输入里有多个问题，优先处理当前显式问的那个。
不要在结尾新增无关问题。
```

再比如：

```text
反馈要具体
```
要转成：

```text
当用户写出伪代码时，点名具体变量、语句或数值变化。
具体指出错误的原本输入内容, 并说明为何错误
不要说“这里没写稳”
```

判断一条规则有没有操作性，可以问：

```text
另一个 agent 看到这条规则，能不能知道下一句话怎么写？
能不能判断自己有没有违反？
能不能拿样本测试？
```

如果不能，它还只是概念。
